# PLAN.md — porting the ska engine to C++/WASM

Goal: a single static HTML page that plays the infinite procedural ska sequence in the
browser, generated by a C++→WASM port of `reference/MusicGen.cs` + `reference/VibeCodec.cs`,
scheduled with Web Audio per `reference/MusicController.cs`. Shareable via `#vibe:tag:n`.

The work splits into: **(A)** get the algorithm running natively in C++ with proven parity,
**(B)** wrap it in WASM, **(C)** build the page + audio scheduler, **(D)** UI + sharing, **(E)**
polish/dist. Do A fully before B — a native build with a parity test is the cheapest place to
catch PRNG drift.

---

## Phase A — C++ core, native, parity-locked

1. **`src/prng.h`** — port `Xmur3` and the `Rng` class.
   - `uint32_t xmur3(const std::string&)`, all ops `unchecked` (natural `uint32_t` wrap).
   - `struct Rng { uint32_t a; float next(); int next_int(int n); bool chance(float p);
     template<T> T pick(span); }`. Match `Next()` math exactly, including the `/ 4294967296f`
     and the `Int(n) = min(n-1, (int)(next()*n))` clamp.
2. **Golden vectors.** Add a tiny C# console snippet (or temporary log in the game) to dump:
   - first 32 `next()` floats for seeds `"rotaliate"`, `"drums:rotaliate"`, `"bd44ac2a:23"`;
   - the composed bpm/scale-index/progression-index/instrument/bass-pattern for a few `tag:n`.
   Store as `test/golden.txt`. **`make test` asserts the C++ matches bit-for-bit.**
3. **`src/music_gen`** — port `MusicGen`:
   - `Config` struct = copy every field + default from `MusicGen.Config` (including the new
     `DrumBusy = 0.6f`, `TripletChance = 0.06f`).
   - Harmony tables (`Scales`, `Progressions`, `BassPatterns`), `Compose()` in the **exact RNG
     draw order**, then the per-bar render calls.
   - Synthesiser: `RenderPatch` (unison osc → optional high-pass → Chamberlin SVF with cutoff
     env → soft drive → AD/sustain env), drum voices, `RenderLead`, ornament/triplet logic.
     Keep `float` where C# uses `float`. Output interleaved stereo `std::vector<float>` (skip
     the int16 round-trip; WebAudio wants float, and it lets us defer the mono downmix).
   - Expose `generate_samples(seed, cfg) -> {pcm, sample_rate}`.
4. **`src/vibe_codec`** — port `VibeCodec`: the `Field` list (name/min/max/int/choices/get/set
   as 0..1), `encode`, `apply`, `looks_like_vibe`, `length`. The list order must equal the C#.
5. `make test` green ⇒ algorithm is correct. Do not move on until it is.

**Risks:** signed/unsigned shifts, `float` vs `double` divergence in `sin/exp/tanh`, and the
two-RNG split (melodic vs `"drums:"` stream). The golden test pins all three.

## Phase B — WASM bindings

1. **`src/bindings.cpp`** with Emscripten. Decide the boundary:
   - `generate(const char* seedStr, const float* cfg, int cfgLen) -> ptr` writing a length-
     prefixed float buffer into WASM heap (JS reads via `HEAPF32`), **or** use `embind` to
     return a `val`-wrapped `Float32Array`. `embind` is less fiddly; raw heap is faster. Start
     with `embind`.
   - Export: `generateSong(seedStr, cfgArray) -> {sampleRate, left:Float32Array,
     right:Float32Array}`, `encodeVibe(cfgArray)->string`, `decodeVibe(str, cfgArray)`,
     `vibeFields() -> [{name,min,max,isInt,choices}]` (drives the UI so JS never hardcodes the
     field list), `parseSeed(str) -> {vibe, tag, n}`.
   - Represent `Config` on the JS side as a plain `Float32Array` in field order + a few extras;
     `bindings.cpp` maps it to/from the C++ `Config`. Keep one canonical ordering.
2. **`Makefile`:**
   ```
   emcc -O3 -std=c++17 -s MODULARIZE=1 -s EXPORT_ES6=1 --bind \
        src/bindings.cpp src/music_gen.cpp src/vibe_codec.cpp \
        -o build/skafinity.js
   ```
   (`-s ALLOW_MEMORY_GROWTH=1` since a 75 s stereo float loop ≈ 19 MB before GC.)
3. Smoke test in a scratch HTML: generate one song, log sampleRate + length.

## Phase C — Web Audio sequencer (`web/app.js`)

Port `MusicController`'s scheduling, not its s&box plumbing.

1. `AudioContext` (created/resumed on the play gesture). Master `GainNode` → destination for
   volume.
2. **Song = AudioBuffer.** `bufFromSong(song)` copies the WASM Float32 channels into a 2-ch
   `AudioBuffer` at `song.sampleRate`.
3. **Look-ahead buffer** (`AheadCount`, default 5). Generate in a **Web Worker** that owns its
   own WASM instance, so rendering never blocks the audio/UI thread. `postMessage(seedStr,
   cfg)` → transfer the channel `ArrayBuffer`s back. Mirror `FillAhead`'s "top up to N, drop
   stale on sequence restart (`seq` token)" logic.
4. **Playback chain.** Maintain a running schedule clock (`ctx.currentTime`-based). For each
   song: schedule `LoopsPerSong` passes; reserve the final `Crossfade` seconds; when the next
   song is ready, schedule its head overlapping the reserved tail with equal-power gain ramps
   (`CrossfadeOverlap` centred), exactly as `PushTransition` computes `W`, `cross`, `ws`,
   `we`. Advance `n`, persist to `localStorage`.
   - Simplest correct approach: schedule each pass as its own `AudioBufferSourceNode` with
     absolute `start(when)`; crossfade by ramping two `GainNode`s. Avoid sample-stitching in JS.
5. **Restart** (`StartSequence`): bump `seq`, stop scheduled nodes, clear look-ahead, re-render
   from current `n`. Debounce vibe-slider edits ~0.35 s like the game.

**Gotcha:** Web Audio scheduling is fire-and-forget in absolute time; keep a small JS timer
that tops up the schedule a few seconds ahead (like the `_pushedSeconds - _sinceStart < 2`
check) rather than scheduling the whole infinity up front.

## Phase D — UI + sharing (`web/index.html` + `style.css`)

1. Transport: Play/Pause, Prev/Next song (`n∓1`), current `vibe:tag:n` shown + Copy.
2. **Seed field**: paste `vibe:tag:n` / `tag:n` / `tag` → `parseSeed` → restart. "Use a word"
   = set tag, clear vibe.
3. **Vibe editor**: build rows from `vibeFields()`. Continuous knobs → `<input type=range>`
   (real sliders! the win over s&box); choice knobs (`LEAD INSTR`) → `<select>`. On input,
   update the `cfg` array, re-encode vibe, push to `location.hash`, debounce-restart.
   - Respect the narrowed ranges from the source (`TRIPLETS` max 0.1, vibe mins/maxes per
     `Field`). Show the `Field.Display` value (percent / int / choice label).
4. **URL as state**: read `location.hash` on load → seed; write it on every change. Reloads and
   shared links reproduce the song. `localStorage` only for "resume where I left off" when the
   hash is empty.
5. **Save WAV**: reuse `WavFromSamples` logic (port the header writer) to download the current
   loop — the game's "save song" feature.

## Phase E — polish / distribution

- `make dist`: inline `skafinity.js` + base64 the `.wasm` into one `skafinity.html` (Emscripten
  `-s SINGLE_FILE=1` does this for you) so the whole toy is one shareable file.
- Visualiser (optional, cheap): an `AnalyserNode` driving a canvas — bouncing bars themed
  like the cabinet. Nice but not required.
- Mobile: one render is ~tens of ms; fine. Watch memory on `AheadCount` — maybe drop to 3 on
  small devices.
- README with a screenshot + a few fun example seeds.

---

## Open decisions (pick during A/B, don't block on them now)

- **Stereo vs mono on web.** The game downmixes to mono for positional 2D playback; the web has
  no such constraint — **keep stereo** (the pan/horn spread is half the vibe). Trivial: just
  don't downmix.
- **`embind` vs raw heap.** Start `embind` for ergonomics; switch the hot `generateSong` path
  to raw `HEAPF32` + a length-prefixed pointer if generation latency matters.
- **Worker count.** One worker with one WASM instance is enough (generation is faster than
  playback). Only shard if look-ahead can't keep up on weak hardware.
- **Exact-audio parity with the game.** Aim for *composition* parity (shared seeds = same
  arrangement) as a hard requirement; treat sample-identical synthesis as best-effort.

---

## Definition of done

Open `skafinity.html`, press play, hear endless ska that crossfades between songs; drag a vibe
slider and the sound changes; paste a `vibe:tag:n` from the game and get the same arrangement;
share the URL and a friend hears the same thing. One file, no server.
